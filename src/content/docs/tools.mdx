Browy ships with the browser-driving tools enabled by default. Any
individual tool can be turned off in **Settings â†’ Tools** for stricter
modes (read-only, no JS evaluation, and so on).

Browy also exposes a small set of **host tools** from the Copilot SDK
(`bash`, `read_file`, `write_file`, `grep`, `glob`, `web_fetch`). These
reach your **machine's filesystem and shell**, not just the browser
tab â€” so they're **off by default** and protected by a strict allowlist.
You can opt in to any of them per-tool in Settings if you trust the
sites you visit. (See [Security model](/docs/security) for the
threat-model details.)

<Aside type="note">
The authoritative list lives in
[`src/agent/tools/browser.ts`](https://github.com/BrowyHQ/browy/blob/main/src/agent/tools/browser.ts).
Search for `register({` to see every tool. The host-tool allowlist is
in [`src/agent/loop.ts`](https://github.com/BrowyHQ/browy/blob/main/src/agent/loop.ts)
(search for `HOST_TOOL_ALLOWLIST`).
</Aside>

## Default vs opt-in

| | Default | Where it runs | Toggle |
|---|---|---|---|
| **Browser tools** (everything below) | ON | The active tab via CDP | Disable per-tool in Settings |
| **Host tools** (`bash`, `read_file`, `write_file`, `grep`, `glob`, `web_fetch`) | OFF | Your machine, with your user privileges | Enable per-tool in Settings â†’ Tools â†’ host (advanced) |

Host tools are gated because a prompt-injection on a visited page could
otherwise read `~/.ssh`, run shell commands, or fetch from internal
URLs. Leave them off unless you actively need agentic file/shell work
and you're driving the agent on pages you trust.

## Configurable per-tool toggles

Every browser tool can be turned off individually in **Settings â†’
Tools** (right-click the Browy toolbar icon â†’ Options). Toggles are
grouped by category â€” _inspect_, _navigate_, _interact_, _devtools_ â€”
and persist in `chrome.storage.local`. Disabled tools are stripped from
the SDK's allowlist and from the function-definition list sent to the
model, so the agent doesn't see they exist.

Common reasons to disable a tool:

<CardGrid>
  <Card title="Stricter sandbox" icon="seti:lock">
    Turn off `evaluate_js` and `run_script` if you don't want the agent
    running JavaScript in the inspected page. The structured tools
    (`click_index`, `type_index`, `navigate`, etc.) keep working.
  </Card>
  <Card title="Read-only mode" icon="seti:eye">
    Disable everything in _interact_ and _navigate_ and keep _inspect_
    on. The agent can answer questions about pages but can't change
    them.
  </Card>
  <Card title="No JS evaluation" icon="seti:javascript">
    Some teams disable `evaluate_js`, `run_script`, and `query_dom` to
    limit the agent to the structured accessibility-tree tools only.
  </Card>
</CardGrid>

Click **enable all** in the toolbar to undo every override at once.
Changes apply on the next chat (or after `/clear`).

## What about host shell, filesystem, web_fetch?

The Copilot SDK includes `bash`, `read_file`, `write_file`,
`web_fetch`, `grep`, and `glob` as standard tools â€” they're how the
`copilot` CLI does coding work in a terminal. Browy's browser surfaces
do **not** expose any of them, by design. The native host registers a
strict allowlist when it opens a Copilot session, so even if a model
tried to call `bash` in response to a prompt-injected page, the SDK
would reject it. There is no Settings toggle to turn them on, and no
in-browser path to opt in.

If you need a terminal coding agent, use the
[GitHub Copilot CLI](https://docs.github.com/en/copilot/github-copilot-in-the-cli)
directly.
